# OSINT//DESK — Ingestion Adapter Framework
## กรอบการเชื่อมต่อแหล่งข่าวภายนอก

เวอร์ชัน 1.0 · มิถุนายน 2568

## 1. ปัญหาที่ Adapter Framework แก้

ห้องข่าวแต่ละแห่งมีแหล่งข่าวต่างกันสิ้นเชิง บางแห่งใช้สายข่าวต่างประเทศอย่าง Reuters หรือ AP บางแห่งมีระบบจัดการเนื้อหาภายในที่พัฒนาเอง บางแห่งพึ่งโซเชียลมีเดียเป็นหลัก และเกือบทุกแห่งรับเบาะแสผ่านอีเมล ไลน์ หรือเทเลแกรม

หากระบบรับข้อมูลได้แค่รูปแบบเดียวที่กำหนดตายตัว การขายให้ห้องข่าวใหม่แต่ละแห่งจะต้องแก้ไขแกนกลางของระบบทุกครั้ง ซึ่งช้า เสี่ยง และไม่คุ้มค่า

Adapter Framework แก้ปัญหานี้ด้วยหลักการง่ายๆ คือแยกส่วนที่ "เชื่อมต่อกับแหล่งข้อมูล" ออกจากส่วนที่ "ประมวลผลข้อมูล" อย่างสิ้นเชิง แต่ละแหล่งข้อมูลมีตัวแปลงของตัวเองที่ทำหน้าที่เดียวคือแปลงข้อมูลดิบให้เป็นรูปแบบมาตรฐานกลาง จากนั้นทั้งระบบประมวลผลข้อมูลมาตรฐานนี้เหมือนกันหมดไม่ว่ามาจากแหล่งใด

ผลลัพธ์คือการเพิ่มแหล่งข้อมูลใหม่ทำได้โดยเขียนตัวแปลงตัวเดียว ไม่ต้องแตะแกนกลางของระบบเลย

## 2. รูปแบบข้อมูลมาตรฐานกลาง

หัวใจของ Adapter Framework คือรูปแบบข้อมูลมาตรฐานที่เรียกว่า Canonical Feed Item ทุกตัวแปลงต้องแปลงข้อมูลจากแหล่งของตนให้อยู่ในรูปแบบนี้ก่อนส่งเข้าระบบ

รูปแบบมาตรฐานประกอบด้วยข้อมูลจำเป็นได้แก่ รหัสเฉพาะของรายการ หัวข้อ เนื้อหา แหล่งที่มา ลิงก์ต้นทาง เวลาที่เผยแพร่ เวลาที่รับเข้าระบบ ประเภทของแหล่ง ภาษา และรายการสื่อที่แนบมา นอกจากนี้ยังมีช่องสำหรับข้อมูลเพิ่มเติมเฉพาะแหล่งที่อาจมีประโยชน์ในการประมวลผลภายหลัง

เมื่อข้อมูลทุกแหล่งถูกแปลงเป็นรูปแบบเดียวกันแล้ว ระบบคัดกรองข่าว ระบบให้คะแนน และระบบเฝ้าระวังทั้งหมดทำงานกับข้อมูลนี้ได้ทันทีโดยไม่ต้องรู้ว่าเดิมมาจากแหล่งใด

---

## 3. ประเภทของตัวแปลง

ระบบรองรับตัวแปลงหกประเภทหลัก แต่ละประเภทออกแบบให้รับมือกับลักษณะเฉพาะของแหล่งข้อมูลนั้น

### 3.1 ตัวแปลง RSS และ Atom

เป็นตัวแปลงพื้นฐานที่สุด รับฟีดมาตรฐานจากเว็บไซต์ข่าวและบล็อกทั่วไป ตัวแปลงนี้ดึงฟีดตามรอบเวลา แยกแต่ละรายการ และแปลงเป็นรูปแบบมาตรฐาน รองรับทั้งฟีดสาธารณะและฟีดที่ต้องยืนยันตัวตน

### 3.2 ตัวแปลงสายข่าว

รองรับสายข่าวระดับมืออาชีพอย่าง Reuters, AP และสำนักข่าวไทยที่มีระบบส่งข่าวเฉพาะ ตัวแปลงประเภทนี้มักต้องจัดการกับรูปแบบข้อมูลเฉพาะของแต่ละสำนัก การยืนยันตัวตนด้วยกุญแจเฉพาะ และการจัดการโควต้าการเรียกใช้ ออกแบบให้รองรับทั้งแบบดึงข้อมูลและแบบรับข้อมูลที่ส่งมา

### 3.3 ตัวแปลงโซเชียลมีเดีย

เชื่อมต่อกับ Facebook, X และ TikTok ผ่านส่วนต่อประสานทางการของแต่ละแพลตฟอร์ม ตัวแปลงประเภทนี้ซับซ้อนกว่าเพราะต้องจัดการกับการยืนยันตัวตน การจำกัดอัตราการเรียกใช้ และการเปลี่ยนแปลงของส่วนต่อประสานบ่อยครั้ง รวมถึงดึงสื่อที่แนบมาเพื่อส่งต่อให้ระบบตรวจสอบ

### 3.4 ตัวแปลงระบบจัดการเนื้อหา

เชื่อมต่อกับระบบจัดการเนื้อหาภายในของห้องข่าวที่พัฒนาเอง ตัวแปลงประเภทนี้ปรับแต่งตามแต่ละลูกค้า เชื่อมผ่านฐานข้อมูลโดยตรง ส่วนต่อประสานภายใน หรือการส่งออกข้อมูล ทำให้ข่าวที่ทีมงานผลิตเองเข้าสู่ระบบคัดกรองได้ด้วย

### 3.5 ตัวแปลงข้อความ

รับเบาะแสและข้อมูลผ่านอีเมล ไลน์ และเทเลแกรม ซึ่งเป็นช่องทางที่ห้องข่าวรับข้อมูลจากแหล่งข่าวและประชาชนทั่วไป ตัวแปลงประเภทนี้แยกเนื้อหาออกจากข้อความ จัดการไฟล์แนบ และส่งต่อให้ระบบประมวลผลพร้อมระบุว่าเป็นข้อมูลที่ยังไม่ยืนยัน

### 3.6 ตัวแปลงเว็บฮุก

เป็นตัวแปลงแบบยืดหยุ่นที่สุด รับข้อมูลที่ระบบภายนอกส่งมาแบบทันที เหมาะกับการเชื่อมต่อกับเครื่องมือเฉพาะที่ลูกค้าใช้อยู่ ตัวแปลงนี้กำหนดรูปแบบการแปลงได้ผ่านการตั้งค่า ทำให้รองรับแหล่งข้อมูลที่ไม่เข้าพวกกับห้าประเภทแรกได้

## 4. วงจรการทำงานของตัวแปลง

ตัวแปลงทุกตัวทำงานตามวงจรเดียวกันสี่ขั้นตอน

ขั้นแรกคือการเชื่อมต่อ ตัวแปลงสร้างการเชื่อมต่อกับแหล่งข้อมูลตามวิธีของแต่ละแหล่ง ทั้งการยืนยันตัวตนและการตั้งค่าที่จำเป็น

ขั้นที่สองคือการดึงข้อมูล ตัวแปลงดึงข้อมูลใหม่ตามรอบเวลาที่กำหนดหรือรับข้อมูลที่ส่งมาทันที พร้อมจดจำว่าดึงถึงจุดไหนแล้วเพื่อไม่ให้ดึงซ้ำ

ขั้นที่สามคือการแปลง ตัวแปลงเปลี่ยนข้อมูลดิบให้เป็นรูปแบบมาตรฐานกลาง พร้อมตรวจสอบความครบถ้วนและกรองข้อมูลที่ไม่สมบูรณ์ออก

ขั้นสุดท้ายคือการส่งต่อ ตัวแปลงส่งข้อมูลมาตรฐานเข้าคิวกลางของระบบ จากจุดนี้ข้อมูลเข้าสู่กระบวนการคัดกรองและประมวลผลเหมือนกันหมดไม่ว่ามาจากแหล่งใด

---

## 5. การจัดการความผิดพลาดและความน่าเชื่อถือ

แหล่งข้อมูลภายนอกไม่เสถียรเสมอไป ระบบจึงออกแบบให้ทนทานต่อความผิดพลาด

หากแหล่งข้อมูลใดล่มหรือตอบช้า ตัวแปลงจะลองใหม่ตามจังหวะที่เหมาะสมโดยไม่กระทบแหล่งอื่น หากยังล้มเหลวต่อเนื่องระบบจะแจ้งเตือนผู้ดูแลแต่ไม่หยุดทำงานทั้งระบบ

ข้อมูลที่แปลงไม่สำเร็จจะถูกเก็บแยกไว้เพื่อตรวจสอบภายหลัง ไม่ทิ้งไปเฉยๆ ทำให้ไม่พลาดข้อมูลสำคัญเพราะปัญหาทางเทคนิคชั่วคราว

ทุกตัวแปลงมีการเฝ้าระวังสุขภาพแยกกัน ผู้ดูแลเห็นสถานะของแต่ละแหล่งได้ว่าทำงานปกติหรือมีปัญหา ดึงข้อมูลล่าสุดเมื่อไหร่ และมีอัตราความสำเร็จเท่าใด

---

## 6. การตั้งค่าผ่านส่วนติดต่อผู้ใช้

ผู้ดูแลระบบเพิ่มและจัดการแหล่งข้อมูลได้ผ่านหน้าตั้งค่าโดยไม่ต้องเขียนโค้ด

สำหรับแหล่งข้อมูลมาตรฐานอย่าง RSS เพียงใส่ที่อยู่ฟีดและกำหนดรอบเวลาดึงข้อมูล สำหรับแหล่งที่ซับซ้อนกว่าอย่างสายข่าวหรือโซเชียลมีเดีย ระบบมีแบบฟอร์มให้กรอกข้อมูลการเชื่อมต่อและการยืนยันตัวตน

ผู้ดูแลทดสอบการเชื่อมต่อได้ก่อนเปิดใช้จริง เห็นตัวอย่างข้อมูลที่ดึงมา และปรับแต่งการแปลงให้เหมาะสม เมื่อพอใจจึงเปิดใช้งานให้ข้อมูลไหลเข้าระบบ

แต่ละแหล่งกำหนดได้ว่าจะให้น้ำหนักในการคัดกรองอย่างไร เช่น ข่าวจากสายข่าวที่เชื่อถือได้อาจได้น้ำหนักสูงกว่าข้อมูลจากโซเชียลมีเดียที่ยังไม่ยืนยัน

---

## 7. ความสำคัญเชิงธุรกิจ

Adapter Framework ไม่ใช่แค่ความสามารถทางเทคนิค แต่เป็นปัจจัยสำคัญที่ทำให้ขายผลิตภัณฑ์ได้

เมื่อระบบรองรับแหล่งข้อมูลได้ทุกแบบโดยไม่ต้องแก้แกนกลาง การติดตั้งให้ลูกค้าใหม่ทำได้เร็วขึ้นมาก ทีมขายสามารถรับปากได้ว่าระบบเชื่อมต่อกับแหล่งข่าวที่ลูกค้าใช้อยู่แล้ว ซึ่งเป็นข้อกังวลแรกของห้องข่าวทุกแห่ง

นอกจากนี้ยังเปิดโอกาสสร้างรายได้เพิ่มจากการพัฒนาตัวแปลงเฉพาะสำหรับแหล่งข้อมูลที่ลูกค้าต้องการเป็นพิเศษ ซึ่งกลายเป็นบริการที่มีมูลค่าในตัวเอง
