## 1. แนวคิดและจุดประสงค์

OSINT//DESK เดิมทำงานสองขั้นตอน ได้แก่ รู้ก่อนและตรวจสอบได้ Scenario Simulation Module เพิ่มขั้นตอนที่สามคือประเมินผลที่น่าจะเกิดขึ้น

เมื่อข่าวผ่านกระบวนการสืบสวนและมีหลักฐานยืนยันแล้ว ระบบส่งข้อมูลนั้นให้ MiroFish เป็น seed เพื่อสร้างโลกจำลองและประเมินว่าเหตุการณ์นี้จะส่งผลอย่างไรต่อสังคม ตลาด ความคิดเห็นสาธารณะ และการเมือง

ผลลัพธ์คือ Impact Assessment ที่แนบท้าย Brief ทุกฉบับ แบ่งเป็นสามสถานการณ์ที่น่าจะเกิด ช่วยให้บรรณาธิการและผู้บริหารห้องข่าวตัดสินใจได้แม่นยำขึ้นว่าจะลงข่าวเมื่อไหร่ นำเสนออย่างไร และควรติดตามอะไรต่อ

## 2. วิธีทำงานของ MiroFish

MiroFish เป็น swarm intelligence engine ที่สร้าง digital world จำลองจาก seed information โดยมีขั้นตอนสำคัญสี่ขั้น

ขั้นแรกคือการสกัด ontology ระบบอ่านเอกสาร seed แล้วสกัดเอนทิตีที่เกี่ยวข้องทั้งหมด ทั้งบุคคล องค์กร เหตุการณ์ และความสัมพันธ์ระหว่างกัน แล้วสร้าง knowledge graph ผ่าน GraphRAG

ขั้นที่สองคือการสร้าง Agent ระบบสร้าง agent หลายร้อยถึงหลายพันตัว แต่ละตัวมีบุคลิกเฉพาะ ความจำระยะยาว และ logic พฤติกรรมของตัวเอง สะท้อนกลุ่มคนในสังคมจริง เช่น นักการเมือง นักธุรกิจ นักข่าว ประชาชนทั่วไป นักวิชาการ และผู้ใช้โซเชียลมีเดีย

ขั้นที่สามคือการรัน simulation ภายในโลกจำลอง agent เหล่านี้ interact กัน แลกเปลี่ยนข้อมูล โต้เถียง ปรับเปลี่ยนความคิด และแพร่กระจายข่าวสาร ผลลัพธ์คือภาพรวมของกระแสที่น่าจะเกิดขึ้นในแต่ละช่วงเวลา

ขั้นสุดท้ายคือการสรุปผล ReportAgent วิเคราะห์ผลจาก simulation แล้วสร้างรายงาน Impact Assessment พร้อมระบุสถานการณ์ที่เป็นไปได้สามแบบ

## 3. การเชื่อมต่อเข้า OSINT//DESK

### 3.1 Trigger

Simulation รันได้สองแบบ แบบแรกคือ manual โดยนักวิเคราะห์กดปุ่ม "จำลองสถานการณ์" จากหน้า Investigation หรือ Brief Builder แบบที่สองคือ auto-trigger เมื่อ triage score สูงกว่าเกณฑ์ที่กำหนด เช่น 14 ขึ้นไป ระบบจะเริ่ม simulation อัตโนมัติในเบื้องหลัง

### 3.2 Seed Preparation

ก่อนส่งให้ MiroFish ระบบประกอบ seed document จากหลักฐานที่ verified แล้วใน Evidence Board ได้แก่ verified findings ทั้งหมด entity relationships จาก Neo4j timeline ของเหตุการณ์ และ context จากผลการค้นหาของ Perplexica

### 3.3 Local Stack Integration

ใช้ fork ของ MiroFish ที่รันแบบ fully local โดยแทนที่ Zep Cloud ด้วย Neo4j ที่มีอยู่แล้ว ใช้ Ollama เป็น LLM backend ผ่าน OpenAI-compatible endpoint และ GraphRAG layer เชื่อมกับ Neo4j instance เดิมของระบบ ทำให้ไม่มีข้อมูลออกนอก server

### 3.4 Output Integration

ผลจาก simulation กลับมาเป็น Impact Assessment JSON ที่ระบบแปลงและแสดงในสี่ที่ ได้แก่ แท็บใหม่ใน Investigation workspace แท็บ Impact ใน Brief Builder ส่วน Scenario Summary ใน Today's Intel สำหรับข่าว Priority และรายงาน PDF ที่ export ได้

## 4. โครงสร้าง Output

Impact Assessment ประกอบด้วยสามส่วนหลัก

ส่วนแรกคือ Scenario Matrix แสดงสถานการณ์สามแบบพร้อมความน่าจะเป็นและ timeline ที่คาดการณ์

ส่วนที่สองคือ Signal Tracking ระบุสัญญาณที่ต้องเฝ้าดูเพื่อรู้ว่าสถานการณ์กำลังเดินไปทางไหน เช่น การเคลื่อนไหวของตลาด ท่าทีของบุคคลสำคัญ หรือกระแสโซเชียล

ส่วนที่สามคือ Recommended Coverage Strategy แนะนำว่าห้องข่าวควรนำเสนอข่าวนี้อย่างไร เมื่อไหร่ และควรติดตามประเด็นใดต่อ

## 5. ข้อจำกัดที่ต้องสื่อสาร

สิ่งสำคัญที่ต้องสื่อสารให้ชัดเจนกับผู้ใช้คือ Impact Assessment เป็นการประเมินความน่าจะเป็น ไม่ใช่คำทำนายที่แม่นยำ ระบบอาจพลาดตัวแปรที่ไม่คาดคิด และความแม่นยำขึ้นกับคุณภาพของหลักฐานที่ป้อนเข้าไป

ผลลัพธ์เหมาะสำหรับใช้เป็นข้อมูลประกอบการตัดสินใจเชิงบรรณาธิการ ไม่ใช่สำหรับเผยแพร่เป็นข่าว

มิติที่สองคือ upsell opportunity Simulation Module เป็นฟีเจอร์ที่มีต้นทุนการประมวลผลสูง เหมาะสำหรับเสนอในแพ็กเกจระดับ Professional ขึ้นไปเป็น premium add-on
